# SciAgent 面试实战准备

这份文档模拟导师面试的真实追问场景。不是概念解释，是"导师问什么、你怎么答"的实战演练。

---

## 一、核心概念速记卡（30秒版）

| 概念             | 一句话解释                               | 类比                                         |
| ---------------- | ---------------------------------------- | -------------------------------------------- |
| Agent            | 能自主决策和执行动作的AI系统，不只是聊天 | ChatGPT只会纸上谈兵，Agent会自己打开电脑去算 |
| ReAct            | 思考→行动→观察的循环框架               | CFD仿真流程：设置→运行→看结果→调参→重跑  |
| Function Calling | 大模型请求调用外部工具的机制             | 考试时在卷子上写"请监考老师帮我查计算器"     |
| System Prompt    | 给大模型的岗位说明书                     | 新员工入职手册，规定工作流程和注意事项       |
| Self-Correction  | 把错误反馈给大模型，让它自行修正         | CFD不收敛时看报错调参数，本质是一样的        |
| 消息列表         | 累积的完整对话历史，每次调API都要发      | 每天换新实习生，必须把之前所有会议记录给他看 |

## 二、追问链实战演练

### 追问链 1：技术深度

**Q: 你说你做了一个 Agent，具体怎么实现的？**

> 整个系统分四层。最底层是代码执行器，用 subprocess 在独立进程中执行大模型生成的 Python 代码，有 60 秒超时保护。第二层是 API 通信层，封装了 DeepSeek API 的调用，处理消息格式转换和 Function Calling 的解析。核心是第三层，ReAct 主循环——维护一个消息列表，每轮把完整对话历史和工具定义发给大模型，根据大模型的返回决定是执行工具还是结束。最上层是 Streamlit 做的 Web 界面。

**Q（追问）: 为什么用 subprocess 而不是直接 exec？**

> 进程隔离。大模型生成的代码是不可控的，如果用 exec 直接在主进程执行，错误代码可能导致主程序崩溃。subprocess 在独立进程中执行，即使代码出了问题，主程序不受影响。这和 CFD 仿真中把求解器和前处理器分开是一个道理。

**Q（追问）: 消息列表为什么要每次都发完整的？**

> 因为大模型 API 是无状态的。每次调用就像找了一个新人，它不记得之前说过什么。所以必须把所有对话历史全部带上。这也是为什么 Agent 的轮数不能太多——消息列表太长会超过上下文窗口限制。

**Q（追问）: 如果消息列表太长怎么办？**

> 有几种策略：一是压缩早期消息（只保留摘要），二是设置最大轮数上限（我设了10轮），三是用滑动窗口只保留最近几轮。在我的项目中，10个测试任务平均只需要1.6轮，消息长度不是问题。但如果扩展到更复杂的任务，消息管理会成为需要解决的工程问题。

### 追问链 2：Prompt Engineering

**Q: 你的 System Prompt 是怎么设计的？**

> 不是一次写好的，是反复迭代出来的。最初大模型会犯几个典型错误：第一，写了代码但不 print 输出结果，导致我们拿不到数据；**第二，用 plt.show() 弹窗但服务器没有显示器，导致卡死， 要用 `plt.savefig('文件名.png')`——意思是"别弹窗，直接把图保存成文件"**。保存文件不需要屏幕，保存完程序就继续往下跑了。；第三，画图时用中文标注，但 matplotlib 默认没有中文字体，显示成方块。每发现一个问题，我就在 Prompt 里加一条约束。Prompt Engineering 不是写一段话，是一个"发现问题→加约束→验证"的工程化迭代过程。

**Q（追问）: Prompt 的约束是硬约束还是软约束？**

> 软约束。**大模型不一定 100% 遵守**。这和传统编程不同——if-else 是确定性的，但 Prompt 里的指令是概率性的。要真正保证行为，需要在代码层面做后处理校验。比如可以在工具层自动检测代码里是否包含 plt.show()，如果有就自动替换掉。

### 追问链 3：评测和科学性

**Q: 你测了 10 个任务全部成功，这个结果可信吗？**

> 当前 benchmark 偏保守。10 个任务都是标准科学计算题，大模型对这类问题本身就比较擅长，所以 10/10 在意料之中。但这也说明测试还没有触及能力边界。后续应设计对抗性测试——故意给错误公式、要求使用冷门库、提出有歧义的需求——来探测 Agent 的失败模式和鲁棒性边界。

**Q（追问）: 多轮迭代的任务是因为出错了吗？**

> 不是。10个任务0次代码执行错误。出现2-3轮的任务（如Rosenbrock优化3轮、线性方程组3轮），是Agent主动分步执行：先计算、再画图、再分析。这是Agent的正面特征——**任务分解能力**，而非出错后的修复。

**Q（追问）: 你怎么判断Agent的输出是否正确？**

> benchmark.json 里每道题都有 ground_truth，包含预期数值和容差范围，还有 verification_code 可以自动验证。比如三次方程的实根应该约等于2.0946（容差0.001），线性方程组的解应该是 x=1, y=-2, z=-2。我通过对比Agent输出和标注答案来判断正确性。

### 追问链 4：未来方向

**Q: 如果让你继续做这个方向，你打算怎么深入？**

> 三个方向：
>
> **工具扩展**：当前只有一个工具（执行Python代码）。可以接入OpenFOAM做CFD仿真、接入文献检索API做自动调研、接入数据库查询实验数据。这样Agent能完成从调研到仿真到分析的完整科研流程。
>
> **多Agent协作**：一个Agent负责文献调研，一个负责建模，一个负责仿真执行，一个负责结果分析。这涉及Agent间的通信协议和任务分配，是当前学术热点。
>
> **结合研究方向**：微通道散热器优化涉及CFD仿真、NSGA-II优化、参数敏感性分析，流程模式化。如果Agent能自主完成"修改参数→重新划网格→运行CFD→收集结果→更新Pareto前沿"的循环，就能大幅加速优化。

**Q（追问）: 你觉得当前Agent最大的局限是什么？**

> 缺乏深度领域知识。当前Agent依赖大模型的通用知识，能处理标准教科书问题。但面对真正的科研问题——比如选择合适的湍流模型（k-epsilon vs k-omega SST）、判断网格是否足够细、识别数值发散的原因——Agent还做不好。这恰好是值得研究的方向：如何通过RAG（检索增强生成）或领域微调让Agent具备专业判断力。

---

### 追问链 5：你的成长

**Q: 你之前没有大模型的基础，做这个项目你学到了什么？**

> 三个层面：
>
> **技术层面**：大模型不是黑箱，而是通过API调用、Prompt控制、工具扩展的可编程系统。Function Calling让大模型从"只能说"变成"能做事"，这是Agent和Chatbot的本质区别。
>
> **工程层面**：Prompt Engineering是工程化的迭代过程。每条约束背后都有一个实际遇到的bug。中文乱码、代码不输出结果，都是测试中发现然后通过Prompt约束解决的。
>
> **研究层面**：我看到了Agent在科学计算中的潜力和局限。标准问题能处理，但需要深度领域知识的问题还做不好。这恰好是值得研究的方向。

## 四、容易踩的坑

1. **不要说"我用了一个框架"** — 这个项目没有用 LangChain 之类的框架，是自己从头实现的 ReAct 循环。这是加分项，说明你理解底层原理。
2. **不要混淆大模型和Agent** — 大模型是Agent的"大脑"，但Agent还包括工具、循环控制、消息管理等。Agent = 大模型 + 工具 + 编排逻辑。
3. **不要说self-correction是"算法"** — 没有专门的纠错算法，就是把错误信息反馈给大模型，利用大模型自身的理解能力来修正。
4. **承认局限性** — 导师喜欢听到你对局限性的思考。"10/10全部成功"不要说成"Agent无所不能"，要说"当前benchmark还不够有挑战性，需要设计更难的测试"。
5. **强调你的领域结合点** — 你不是随便做了一个Agent，你是把Agent应用在了你擅长的微通道散热器/多目标优化场景。这个交叉点是你的独特优势。

## 五、一分钟电梯演讲（开场用）

> 我做了一个科学计算智能体 SciAgent。用户用自然语言描述问题，比如"优化微通道散热器的宽度和深度，同时最小化热阻和压降"，Agent 基于 ReAct 框架自动完成问题分析、代码生成、执行、纠错和可视化的全流程。核心机制是 Function Calling——大模型通过结构化请求调用代码执行工具，执行结果反馈回大模型形成闭环。我在10个科学计算任务上测试，全部一次成功。项目没有用任何现成的Agent框架，ReAct循环是从头实现的，这让我深入理解了Agent的底层运作机制。
