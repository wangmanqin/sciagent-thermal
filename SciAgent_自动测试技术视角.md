## 一、项目概述

SciAgent 是一个**自然语言驱动的科学计算智能体**。用户用自然语言描述工程计算问题（例如"计算矩形微通道散热器的热阻和压降"），Agent 自动完成：问题分析 -> 代码生成 -> 执行 -> 结果验证 -> 可视化输出，全程无需人工干预。

**核心技术**：基于 ReAct（Reasoning + Acting）推理框架，Agent 在"推理-行动-观察"的闭环中迭代，并具备运行时错误自动检测与自修复能力。

```
用户（自然语言）→ Agent 闭环（ReAct）→ 最终结果 + 可视化
                      ↕            ↕
                  大模型 API      代码沙箱执行器
                 (DeepSeek)     (subprocess 隔离)
```

## 二、与自动测试技术的深度关联

这个项目的一大核心工作，就是为 AI Agent 构建了一套**完整的自动化测试与评测框架**。以下从自动测试技术课程的几个核心概念出发，说明两者的对应关系。

### 2.1 被测系统（SUT）：AI Agent 本身

| 传统自动测试 | 本项目 |
|:---:|:---:|
| 被测设备/传感器/电路 | AI 科学计算 Agent |
| 输入激励信号 | 自然语言问题描述 |
| 输出响应信号 | Agent 生成的代码、计算结果、图表 |
| 测试标准/规范 | 解析解 / 数值标准答案（Ground Truth）|

**关键洞察**：传统自动测试技术面对的是硬件系统（传感器、仪器），输出是确定性的电信号；而本项目面对的是 AI 系统，输出是非确定性的自然语言 + 代码。这要求我们设计**更复杂的测试判定机制（Test Oracle）**来评判"对不对"。

### 2.2 测试用例设计 -- 从课程理论到工程实践

**自动测试技术课程**教我们设计测试用例要考虑：覆盖率、边界条件、难度分级、可重复性。

本项目的 Benchmark 设计**完整遵循了这些原则**：

| 测试设计原则 | 本项目实践 |
|---|---|
| **分级测试** | 10 道题按 Easy -> Medium -> Hard 递进 |
| **功能覆盖** | 覆盖 7 个子领域：基础传热、流体力学、热阻网络、参数分析、单目标优化、多目标优化、综合设计 |
| **边界覆盖** | 包含物理公式歧义场景（Fanning friction factor vs Poiseuille number，数值差 4 倍）|
| **端到端覆盖** | 从单一公式计算到完整设计优化链路 |
| **可重复性** | 每道题有确定的解析解/数值标准答案，支持反复回归测试 |

```
测试覆盖矩阵：

难度\类别   基础传热  流体力学  热阻网络  参数分析  单目标优化  多目标优化  综合设计
Easy         #1,#2     #3                                                    #8
Medium                           #4,#5      #6        #7
Hard                                                              #9         #10
```

### 2.3 自动化测试判定（Test Oracle）-- 本项目的核心创新

**这是与自动测试技术课程最直接相关的部分。**

传统自动测试中，判定被测系统输出是否合格，需要设计**测试判定准则**。对于测控系统，这通常是"测量值是否在容差范围内"。

本项目面对的挑战更大：AI Agent 的输出是**非结构化的自然语言 + 代码**，如何自动判定"对不对"？

我们设计了**三维度自动评分体系**：

```
┌─────────────────────────────────────────────────────┐
│                自动评分系统 (auto_score)              │
│                                                      │
│  维度1: 程序化验证 (40分)                             │
│  ├── 执行预先编写的验证代码（assert 断言）             │
│  ├── 检查: Agent 用的公式逻辑是否正确                  │
│  └── 类比: 自动测试中的"功能测试"                      │
│                                                      │
│  维度2: 数值容差匹配 (40分)                           │
│  ├── 正则提取 Agent 输出中的数值                       │
│  ├── 与期望值做相对容差/绝对容差比较                    │
│  ├── 容差范围: ±3% ~ ±10%（按题目精度要求设定）        │
│  └── 类比: 自动测试中的"精度/容差检验"                 │
│                                                      │
│  维度3: 可视化完整性 (20分)                           │
│  ├── 检测是否生成了 PNG 图表文件                       │
│  └── 类比: 自动测试中的"输出完整性检查"                │
│                                                      │
│  综合得分 = 三维度加权求和 / 满分                      │
│  通过阈值: ≥ 60分                                     │
└─────────────────────────────────────────────────────┘
```

**与传统自动测试容差检验的对比**：

```python
# 传统自动测试：传感器标定
if abs(measured_value - reference_value) <= tolerance:
    PASS

# 本项目：AI Agent 输出验证
# 1. 先从非结构化文本中提取数值（这是额外的挑战）
numbers = extract_numbers_from_text(agent_output)
# 2. 再做容差比较
for expected_val in ground_truth:
    if any(abs(n - expected_val)/expected_val < tolerance_pct for n in numbers):
        PASS
```

### 2.4 测试执行自动化 -- 一键评测全流程

就像自动测试系统可以一键完成"激励-采集-判定-报告"，本项目的评测脚本 `evaluate.py` 实现了**端到端的全自动评测流程**：

```
python evaluate.py
    │
    ├── 加载 benchmark.json（10 道测试用例）
    │
    ├── 逐题运行 ──┬── 向 Agent 发送自然语言问题
    │              ├── Agent 自动推理、生成代码、执行
    │              ├── 收集 Agent 的输出（文本 + 代码执行结果）
    │              └── 三维度自动评分
    │
    ├── 生成评测报告（JSON + Markdown）
    │    ├── 逐题得分 & 评分详情
    │    ├── 通过率、平均分统计
    │    └── 失败原因诊断
    │
    └── 支持多种运行模式
         ├── --ids 1 3 5     选择性测试
         ├── --verify-only   验证标准答案自身的正确性
         └── --dry-run       预览测试计划
```

**关键点**：`--verify-only` 模式是对**标准答案本身的校验** -- 正如自动测试中需要定期校准基准源一样，我们也需要确保 Ground Truth 没有错误。

### 2.5 测试结果与能力边界分析

评测结果揭示了 AI Agent 的**能力边界**，这与自动测试中的"误差分析"和"不确定度评定"理念一致：

| 评测发现 | 测试技术类比 |
|---|---|
| Easy 类平均 96.7 分 | 在标准工况下，被测系统精度达标 |
| 公式歧义场景得分 68 分 | 在边界/极端工况下，系统出现系统误差 |
| 误差原因：Fanning vs Poiseuille（差 4 倍）| 类似于量程选择错误导致的系统偏差 |
| 改进方案：RAG 检索公式库 | 类似于引入标准源校准来消除系统误差 |

## 三、技术架构亮点

### 3.1 Agent 的自修复机制 -- "闭环控制"思想

Agent 的自我纠错能力，本质上是一个**闭环反馈控制系统**：

```
                  ┌──────────────────────────┐
                  │                          │
 问题描述 ──→ [大模型推理] ──→ [代码生成] ──→ [沙箱执行]
                  ↑                              │
                  │         反馈回路              │
                  └──── [错误信息/执行结果] ←──────┘
```

- **正常路径**：推理 -> 生成代码 -> 执行成功 -> 输出结果
- **纠错路径**：推理 -> 生成代码 -> 执行报错 -> 错误信息反馈给大模型 -> 重新推理修正 -> 再次执行
- 最多 10 轮迭代（防止"振荡"不收敛）
- 60 秒执行超时（防止"锁死"）

这与自动测试系统中的**闭环校准**思想完全一致：测量 -> 比较 -> 调整 -> 再测量。

### 3.2 沙箱执行 -- 测试环境隔离

Agent 生成的代码在**独立子进程（subprocess）**中执行，实现了：
- **进程隔离**：Agent 生成的代码崩溃不影响主系统
- **超时保护**：60 秒硬限制，防止死循环
- 这与自动测试系统中**被测设备与测试控制器隔离**的工程实践一致

## 五、总结：为什么这个项目与自动测试技术相关？

1. **测试对象的演进**：从测试硬件（传感器/电路）到测试 AI 系统 -- 测试方法论是通用的，但判定机制需要创新
2. **核心方法论一致**：测试用例分级设计、覆盖率分析、容差判定、自动化执行、报告生成 -- 这些自动测试课程的核心概念在本项目中全部得到了工程实践
3. **闭环控制思想**：Agent 的自修复机制与自动测试系统的闭环校准一脉相承
4. **先进计算技术融合**：将大语言模型（LLM）技术与传统的自动化测试评估框架结合，是"AI + 测控"的一个具体实践案例

## 六、现场演示要点

如果需要演示，建议按以下顺序：

1. **先展示 Web 界面**（`streamlit run app.py`）：输入一个简单问题，看 Agent 实时推理、写代码、执行、输出结果的全过程 -- 体现"自动化"
2. **展示评测脚本**（`python evaluate.py --dry-run`）：看 10 道测试题的设计 -- 体现"测试用例设计"
3. **运行单题评测**（`python evaluate.py --ids 1`）：看自动评分过程 -- 体现"自动判定"
4. **展示评测报告**（`eval_reports/` 目录）：看完整评测结果 -- 体现"测试报告生成"
5. **讲解失败案例**（第 4 题 Fanning vs Poiseuille 问题）：体现"边界测试"和"误差分析"
